## 1. Avazu — Feature Hashing Puro (FL-compatible)

### Problema
O pré-processamento do Avazu usava **frequency encoding** para as colunas categóricas. Essa técnica exige calcular a frequência de cada valor *em todo o dataset*, o que viola a premissa central do Federated Learning:

> Cada nó só enxerga seus próprios dados — não existe passagem global.

No FL, o `client_0` treina com seu shard. Se o frequency encoding foi calculado sobre o dataset completo, o "mapeamento" já inclui informação privada dos outros clientes.

### Solução
Substituído completamente por **Feature Hashing com 1024 dimensões** — uma transformação puramente determinística e algébrica:

```python
hasher = FeatureHasher(n_features=1024, input_type="string")
```

- **Data-independent**: o mapeamento hash é definido pela função matemática, não pelos dados observados.
- **Sem coordenação entre nós**: cada cliente aplica a mesma função sem precisar se comunicar.
- **Mais capacidade**: 1024d vs. 256d anteriores compensam a perda de ordenação por frequência.

### Features finais do Avazu
| Grupo | Quantidade | Origem |
|---|---|---|
| Temporais (hora do dia, dia da semana, período, sin/cos) | 5 | Data-independent |
| Feature Hashing de 21 colunas categóricas | 1024 | Data-independent |
| **Total** | **1029** | |

### Arquivos modificados
- `tools/prepare_datasets.py` — função `prepare_avazu()` reescrita
- `datasets/paths.py` — DATASET_INFO["avazu"] atualizado
- `datasets/avazu.py` — docstring atualizada (256d → 1024d, data-independent)

## 3. Auditoria Completa do Código

Leitura de todos os 30+ arquivos fonte do projeto. Três bugs encontrados e corrigidos:

### Bug 1 — `datasets/higgs_full.py` ausente (crítico)
O arquivo `higgs_full.py` não existia no repositório — apenas um `.pyc` estale de `./__pycache__/`. O import estava comentado em `datasets/__init__.py`.

**Correção**: criado `datasets/higgs_full.py` do zero com a implementação completa (carregamento de `.npy`, particionamento estratificado, suporte a `role="server"` e `role="client"`), e descomentado o import em `__init__.py`.

### Bug 2 — Numeração duplicada em `sdn_bagging.py` (cosmético)
Dois passos numerados como `# 4.` no método `configure_fit()`. O segundo era o passo 5 (seleção dos melhores candidatos).

### Bug 3 — Import de `higgs_full` comentado
Mesmo com o arquivo criado, o import em `datasets/__init__.py` estava comentado, impedindo o auto-registro no `DatasetRegistry`.

## 4. Warm Start — Decaimento Exponencial Gradual

### Problema
O warm start usava uma fórmula **linear** para reduzir `n_estimators`:

```python
reduction = max(0.1, 1.0 - (server_round - 1) * 0.2)
```

Com `LOCAL_EPOCHS = 100` e `NUM_ROUNDS = 20`:
- Round 6+: **apenas 10 árvores** (piso agressivo atingido cedo)
- `learning_rate` permanecia fixo em `0.10` indefinidamente

**Resultado**: a partir do round 6, o modelo praticamente parava de aprender dados novos, e o LR fixo causava risco de overfitting nos ajustes tardios sobre um modelo já bem treinado.

### Solução: decaimento exponencial sincronizado

Adicionadas 4 constantes em `config.py`:

```python
WARM_START_TREE_DECAY     = 0.85   # fator por round para n_estimators
WARM_START_TREE_MIN_RATIO = 0.30   # piso: 30% de LOCAL_EPOCHS

WARM_START_LR_DECAY       = 0.93   # fator por round para learning_rate
WARM_START_LR_MIN_RATIO   = 0.40   # piso: 40% do LR base
```

Fórmula para cada round com warm start:

```
tree_ratio = max(TREE_MIN_RATIO,  TREE_DECAY ^ (server_round - 1))
lr         = max(base_lr * LR_MIN_RATIO,  base_lr * LR_DECAY ^ (server_round - 1))
```

### Tabela de evolução (LOCAL_EPOCHS=100, LR=0.10)

| Round | n_new | lr |
|---|---|---|
| 1 | 100 | 0.100 |
| 2 | 85 | 0.093 |
| 3 | 72 | 0.086 |
| 5 | 52 | 0.075 |
| 7 | 37 | 0.064 |
| 9 | **30** ← piso | 0.056 |
| 15 | 30 | **0.040** ← piso |
| 20 | 30 | 0.040 |

Comparação com o modelo anterior:

| Round | n_new antigo | n_new novo |
|---|---|---|
| 6 | **10** | 44 |
| 9 | 10 | **30** (piso mínimo razoável) |
| 20 | 10 | 30 |

### Implementação
Nova função `_compute_warm_start_schedule()` em `models/factory.py`:
- Centraliza o cálculo para os 3 modelos (XGBoost, LightGBM, CatBoost)
- Respeita `extra_params["learning_rate"]` como base (grid search tunado)
- Aplica o decay sobre o LR tunado, não sobre o default
